---
title: Virtual Infrastructure
description: noumen uses abstract filesystem and computer interfaces that can be backed by local Node.js or remote containers.
---

noumen separates the agent logic from the execution environment through two interfaces: `VirtualFs` for file operations and `VirtualComputer` for shell commands. This lets you run the same agent locally during development and in sandboxed containers in production.

## VirtualFs

The filesystem interface covers file and directory operations:

```typescript
interface VirtualFs {
  readFile(path: string, opts?: ReadOptions): Promise<string>;
  writeFile(path: string, content: string): Promise<void>;
  appendFile(path: string, content: string): Promise<void>;
  deleteFile(path: string, opts?: { recursive?: boolean }): Promise<void>;
  mkdir(path: string, opts?: { recursive?: boolean }): Promise<void>;
  readdir(path: string, opts?: { recursive?: boolean }): Promise<FileEntry[]>;
  exists(path: string): Promise<boolean>;
  stat(path: string): Promise<FileStat>;
}
```

## VirtualComputer

The computer interface handles shell command execution:

```typescript
interface VirtualComputer {
  executeCommand(command: string, opts?: ExecOptions): Promise<CommandResult>;
}
```

Where `ExecOptions` supports `cwd`, `timeout`, and `env`, and `CommandResult` returns `exitCode`, `stdout`, and `stderr`.

## Local (Node.js)

For local development, use the Node.js-backed implementations:

```typescript
import { LocalFs, LocalComputer } from "noumen";

const fs = new LocalFs({ basePath: "/my/project" });
const computer = new LocalComputer({ defaultCwd: "/my/project" });
```

`LocalFs` uses `node:fs/promises` under the hood. All paths are resolved relative to `basePath`, which acts as a security boundary -- the agent cannot access files outside it.

`LocalComputer` uses `node:child_process.exec` with a configurable default working directory, timeout, and maximum output buffer size.

### LocalFs options

| Option | Type | Default | Description |
|--------|------|---------|-------------|
| `basePath` | `string` | required | Root directory for all file operations |

### LocalComputer options

| Option | Type | Default | Description |
|--------|------|---------|-------------|
| `defaultCwd` | `string` | required | Default working directory for commands |
| `timeout` | `number` | `30000` | Command timeout in milliseconds |
| `maxBuffer` | `number` | `10 * 1024 * 1024` | Maximum stdout/stderr buffer size |

## sprites.dev (Remote containers)

For sandboxed execution in remote [sprites.dev](https://docs.sprites.dev) containers:

```typescript
import { SpritesFs, SpritesComputer } from "noumen";

const fs = new SpritesFs({
  token: process.env.SPRITE_TOKEN!,
  spriteName: "my-sprite",
});

const computer = new SpritesComputer({
  token: process.env.SPRITE_TOKEN!,
  spriteName: "my-sprite",
});
```

The Sprites implementations use the sprites.dev HTTP API. The agent runs against a remote filesystem and shell, providing full isolation.

### SpritesFs options

| Option | Type | Description |
|--------|------|-------------|
| `token` | `string` | sprites.dev API token |
| `spriteName` | `string` | Name of the sprite container |
| `baseURL` | `string` | API base URL (optional) |

### SpritesComputer options

| Option | Type | Description |
|--------|------|-------------|
| `token` | `string` | sprites.dev API token |
| `spriteName` | `string` | Name of the sprite container |
| `baseURL` | `string` | API base URL (optional) |

## Custom implementations

You can implement `VirtualFs` and `VirtualComputer` for any backend -- Docker containers, cloud VMs, in-memory test environments, or anything else. The interfaces are intentionally minimal to make this straightforward.
